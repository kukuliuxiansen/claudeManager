tdd-workflow

测试驱动开发工作流，强制执行先写测试的方法论

类型: Testing Skill

实现效果

tdd-workflow Skill 是测试驱动开发(TDD)的专业工作流技能，强制执行"先写测试，后写代码"的开发方法论。确保代码覆盖率 80% 以上，提升代码质量和可维护性。

🔴 RED → 🟢 GREEN → ♻️ REFACTOR → ✅ VERIFY

  • 强制 TDD 流程

    确保每个功能都先编写测试用例，再实现功能代码

  • 覆盖率保证

    自动验证测试覆盖率，确保达到 80% 以上的代码覆盖

  • 重构引导

    在测试通过后，引导进行安全的代码重构优化

  • 持续验证

    每次修改后自动运行测试，验证代码正确性

    • 核心原理

    TDD 的核心原理是红-绿-重构循环。先写失败的测试(RED)，再写最少代码使测试通过(GREEN)，最后重构优化代码(REFACTOR)。这种工作流确保代码始终可测试，并且有完整的测试覆盖。

    TDD 三定律:

    • 第一定律: 在编写失败的单元测试之前，不编写产品代码
    • 第二定律: 只编写足以失败的测试，不编写更多的测试
    • 第三定律: 只编写足以通过当前失败测试的产品代码

    收益:

    • 更高的代码质量和更少的 Bug
    • 更好的代码设计和解耦
    • 完整的测试文档
    • 更自信的重构能力

    使用方法

    tdd-workflow Skill 在编写新功能、修复 Bug 或重构代码时主动触发：

    触发场景与调用 
    // 主动触发条件
1. 编写新功能 - 先写测试用例
2. 修复 Bug - 先写复现测试
3. 重构代码 - 确保测试覆盖
4. 代码审查 - 验证测试质量

// 通过 Skill 工具调用
/tdd-workflow [feature-name]

// 示例：开发用户认证功能
/tdd-workflow user-authentication

// TDD 工作流示例
1. RED: 编写登录失败测试
   describe('UserAuth', () => {
     it('should fail with invalid credentials', () => {
       expect(auth.login('wrong', 'creds')).rejects.toThrow()
     })
   })

2. GREEN: 实现最少代码使测试通过
   async login(username, password) {
     if (username !== 'valid' || password !== 'pass') {
       throw new Error('Invalid credentials')
     }
     return { token: 'xxx' }
   }

3. REFACTOR: 优化代码结构
   async login(username, password) {
     const user = await this.validate(username, password)
     return this.generateToken(user)
   }

    使用技巧

    1

    小步快跑 - 每次只测试一个行为，保持测试简单

    2

    命名清晰 - 测试名称应描述预期行为

    3

    快速反馈 - 测试应该快速运行，便于频繁执行

    4

    覆盖边界 - 测试正常、边界和异常情况

## GREEN Phase
### Implementation
```javascript
class UserAuth {
  async login(username, password) {
    const user = await this.db.findUser(username)
    if (!user || !this.verify(password, user.password)) {
      throw new Error('Invalid credentials')
    }
    return this.generateToken(user)
  }
}
```
