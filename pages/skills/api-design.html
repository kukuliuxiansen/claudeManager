API设计模式

REST API设计模式，包括资源命名、状态码、分页、过滤、错误响应和版本控制

类型: 最佳实践

实现效果

应用API设计模式，确保API的一致性、可维护性和可扩展性。

  • 资源命名规范
    使用名词复数、kebab-case的URL结构，避免动词，建立清晰的资源层次
  • 语义化状态码
    正确使用HTTP状态码，200成功、201创建、400错误、401未授权、404未找到等
  • 统一响应格式
    标准化成功和错误响应结构，包含data、meta、error等字段，便于客户端处理
  • 灵活的分页策略
    支持Offset和Cursor两种分页方式，根据数据量和场景选择合适的方案

    • 核心原理

    URL设计原则

    资源是名词、使用复数、kebab-case命名。子资源通过路径层次表示，动作通过POST请求实现。

    HTTP方法语义

    GET获取、POST创建、PUT替换、PATCH更新、DELETE删除。注意幂等性和安全性属性。

    响应设计

    成功响应包含data字段，集合响应包含meta分页信息，错误响应包含code、message、details。

    版本控制

    推荐URL路径版本控制（/api/v1/），保持向后兼容，维护最多两个活跃版本。

    使用方法

    触发场景与调用 
    # 适用场景
- 设计新的API端点
- 审查现有API契约
- 添加分页、过滤或排序
- 实现API错误处理
- 规划API版本策略

# URL设计示例
GET    /api/v1/users           # 列表
GET    /api/v1/users/:id       # 详情
POST   /api/v1/users           # 创建
PUT    /api/v1/users/:id       # 替换
PATCH  /api/v1/users/:id       # 更新
DELETE /api/v1/users/:id       # 删除

# 分页查询
GET /api/v1/users?page=2&per_page=20
GET /api/v1/users?cursor=xxx&limit=20

    使用技巧

    1
    使用schema验证所有输入，如Zod、Pydantic，确保请求数据的有效性
    2
    大数据集使用Cursor分页，小数据集使用Offset分页，根据场景选择
    3
    错误响应包含code和message字段，便于客户端程序化处理错误
    4
    配置Rate Limiting并返回X-RateLimit-*头部信息，保护API免受滥用

    输出示例

    API响应示例 
    # 成功响应
{
  "data": {
    "id": "abc-123",
    "email": "alice@example.com",
    "name": "Alice"
  }
}

# 集合响应（带分页）
{
  "data": [...],
  "meta": {
    "total": 142,
    "page": 1,
    "per_page": 20,
    "total_pages": 8
  }
}

# 错误响应
{
  "error": {
    "code": "validation_error",
    "message": "Request validation failed",
    "details": [
      { "field": "email", "message": "Invalid email", "code": "invalid_format" }
    ]
  }
}
