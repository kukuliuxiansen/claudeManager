iterative-retrieval

迭代检索模式，解决子代理上下文问题，通过多轮检索逐步完善信息收集

类型: Agent Skill

实现效果

iterative-retrieval 是解决多代理工作流中"上下文问题"的模式，子代理在开始工作时不知道需要什么上下文，通过迭代检索逐步完善。

DISPATCH → EVALUATE → REFINE → LOOP

  • DISPATCH 阶段

    初始广泛查询，收集候选文件，从高层意图开始

  • EVALUATE 阶段

    评估检索内容的相关性，识别缺失上下文

  • REFINE 阶段

    根据评估更新搜索条件，添加新关键词和模式

  • LOOP 阶段

    最多3轮迭代，足够上下文时提前返回

    • 核心原理

    子代理在启动时上下文有限，不知道哪些文件包含相关代码、代码库使用什么模式或术语。迭代检索通过4阶段循环解决这一问题。

    问题:

    • 发送全部: 超出上下文限制
    • 不发送: 代理缺乏关键信息
    • 猜测需要: 经常出错

    相关性评分标准:

    • 高 (0.8-1.0): 直接实现目标功能
    • 中 (0.5-0.7): 包含相关模式或类型
    • 低 (0.2-0.4): 间接相关
    • 无 (0-0.2): 不相关，排除

    使用方法

    iterative-retrieval 适用于启动需要代码库上下文的子代理、构建上下文逐步完善的多代理工作流等场景。

    触发场景与实现 
    // 主动触发条件
1. 启动需要代码库上下文的子代理
2. 构建上下文逐步完善的多代理工作流
3. 遇到"上下文太大"或"缺失上下文"错误
4. 设计代码探索的类 RAG 检索管道
5. 优化代理编排中的 token 使用

// 迭代检索实现
async function iterativeRetrieve(task, maxCycles = 3) {
  let query = createInitialQuery(task);
  let bestContext = [];

  for (let cycle = 0; cycle < maxCycles; cycle++) {
    const candidates = await retrieveFiles(query);
    const evaluation = evaluateRelevance(candidates, task);

    // 检查是否有足够上下文
    const highRelevance = evaluation.filter(e => e.relevance >= 0.7);
    if (highRelevance.length >= 3 && !hasCriticalGaps(evaluation)) {
      return highRelevance;
    }

    // 优化并继续
    query = refineQuery(evaluation, query);
    bestContext = mergeContext(bestContext, highRelevance);
  }

  return bestContext;
}

// 示例: Bug修复上下文
Task: "Fix the authentication token expiry bug"

Cycle 1:
  DISPATCH: 搜索 "token", "auth", "expiry"
  EVALUATE: auth.ts (0.9), tokens.ts (0.8), user.ts (0.3)
  REFINE: 添加 "refresh", "jwt" 关键词

Cycle 2:
  DISPATCH: 搜索优化后的术语
  EVALUATE: session-manager.ts (0.95), jwt-utils.ts (0.85)
  REFINE: 足够上下文 (2个高相关文件)

Result: auth.ts, tokens.ts, session-manager.ts, jwt-utils.ts

    使用技巧

    1

    从宽到窄 - 不要过度指定初始查询，逐步缩小

    2

    学习术语 - 第一轮通常揭示命名约定

    3

    跟踪缺失 - 明确的差距识别驱动优化

    4

    足够即止 - 3个高相关文件优于10个平庸的

    5

    果断排除 - 低相关文件不会变得相关
