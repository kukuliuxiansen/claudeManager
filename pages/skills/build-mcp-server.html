构建MCP服务器

MCP服务器开发的入口技能，确定部署模型、工具设计模式和框架选择

类型: MCP开发

实现效果

通过系统化的问答流程确定MCP服务器的最佳架构和实现路径。

  • 用例探查阶段
    确定连接目标、用户群体、操作数量、UI需求、认证方式，避免选错方向导致重写
  • 部署模型推荐
    远程HTTP（默认推荐）、MCP应用、MCPB、本地stdio，根据场景选择最优路径
  • 工具设计模式
    小操作空间用一工具一操作，大API用搜索+执行模式，控制上下文窗口大小
  • 技能移交
    确定方案后移交到build-mcp-app或build-mcpb等专门技能完成具体实现

    • 核心原理

    部署模型选择

    远程HTTP：零安装摩擦，适用于云API包装。MCPB：需访问本地机器时使用。本地stdio：仅用于原型。

    工具设计原则

    工具schema直接进入Claude上下文窗口，设计比预期更重要。小操作空间用专用工具，大API用搜索+执行保持上下文精简。

    框架选择

    官方TypeScript SDK：最佳spec覆盖，首选。FastMCP 3.x：Python偏好或包装Python库时使用。

    其他原语

    Resources：暴露可浏览上下文。Prompts：斜杠命令工作流。Elicitation：工具中间请求用户输入。Sampling：工具逻辑中需要LLM推理。

    使用方法

    决策矩阵 
    # 场景 -> 部署模型 -> 工具模式
| Scenario                           | Deployment    | Tool pattern     |
|------------------------------------|---------------|------------------|
| 包装小型SaaS API                   | Remote HTTP   | One-per-action   |
| 包装大型SaaS API (50+ endpoints)   | Remote HTTP   | Search + execute |
| SaaS API需丰富表单/选择器          | MCP app       | One-per-action   |
| 驱动本地桌面应用                   | MCPB          | One-per-action   |
| 本地桌面应用+聊天UI                | MCP app(MCPB) | One-per-action   |
| 读写本地文件系统                   | MCPB          | Depends on size  |
| 个人原型                           | Local stdio   | Whatever fastest |

# 探查问题
1. 连接什么？云API -> 远程HTTP，本地进程/文件 -> MCPB
2. 谁使用？仅自己 -> stdio可接受，分发给他人 -> 远程HTTP/MCPB
3. 多少操作？<15 -> 专用工具，大量 -> 搜索+执行
4. 需要UI？简单输入 -> Elicitation，复杂UI -> MCP app
5. 什么认证？API Key -> 简单，OAuth -> 需远程服务器

    使用技巧

    1
    不要在收集完Phase 1答案前开始脚手架，避免选错方向导致痛苦重写
    2
    远程HTTP是默认推荐，除非服务器必须触碰用户本地机器才考虑MCPB
    3
    Elicitation是spec原生的零UI代码方案，先用它评估，不够再升级到Widget
    4
    移交时用一段话重述设计简报，让下一个技能无需重复询问
