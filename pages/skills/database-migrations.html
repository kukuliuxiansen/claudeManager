Database Migrations

数据库迁移最佳实践，支持 schema 变更、数据迁移、零停机部署

类型: Backend Pattern

实现效果

Database Migrations Skill 是数据库模式变更的专业模式，确保安全、可逆、零停机的 schema 变更。

  • 安全迁移原则

    每个变更都是迁移文件，生产环境永不手动修改数据库

  • Schema/Data 分离

    DDL 和 DML 永远分开，避免长事务和难以回滚的问题

  • Expand-Contract 模式

    零停机迁移策略：扩展-迁移-收缩三阶段部署

  • 多 ORM 支持

    Prisma、Drizzle、Kysely、Django、golang-migrate 等

    • 核心原理

    五个核心原则:

    • 每个变更都是迁移 - 永不手动修改生产数据库
    • 生产环境仅向前迁移 - 回滚使用新的向前迁移
    • Schema 和数据迁移分离 - 永不混合 DDL 和 DML
    • 生产数据测试 - 100 行有效的迁移可能在 10M 行时锁定
    • 迁移不可变 - 部署后永不编辑已运行的迁移

    迁移安全检查清单:

    • 有 UP 和 DOWN (或明确标记不可逆)
    • 大表无全表锁 (使用并发操作)
    • 新列有默认值或可空 (永不添加无默认值的 NOT NULL)
    • 索引并发创建 (CREATE INDEX CONCURRENTLY)

    使用方法

    PostgreSQL 安全迁移模式 
    # 添加列安全方式
-- GOOD: 可空列，无锁
ALTER TABLE users ADD COLUMN avatar_url TEXT;

-- GOOD: 带默认值 (Postgres 11+ 即时，无重写)
ALTER TABLE users ADD COLUMN is_active BOOLEAN NOT NULL DEFAULT true;

-- BAD: 现有表上无默认值的 NOT NULL (需要全表重写)
ALTER TABLE users ADD COLUMN role TEXT NOT NULL;

# 无停机添加索引
-- BAD: 阻塞大表写入
CREATE INDEX idx_users_email ON users (email);

-- GOOD: 非阻塞，允许并发写入
CREATE INDEX CONCURRENTLY idx_users_email ON users (email);

# 大数据迁移批处理
DO $$
DECLARE
  batch_size INT := 10000;
  rows_updated INT;
BEGIN
  LOOP
    UPDATE users SET normalized_email = LOWER(email)
    WHERE id IN (
      SELECT id FROM users WHERE normalized_email IS NULL
      LIMIT batch_size FOR UPDATE SKIP LOCKED
    );
    GET DIAGNOSTICS rows_updated = ROW_COUNT;
    EXIT WHEN rows_updated = 0;
    COMMIT;
  END LOOP;
END $$;

    使用技巧

    1

    Expand-Contract 策略 - 零停机重命名：添加新列 → 回填数据 → 更新代码 → 删除旧列

    2

    先删代码再删列 - 先部署移除列引用的代码，再在下一个迁移中删除列

    3

    CONCURRENTLY 无法在事务中 - 大多数迁移工具需要特殊处理并发索引创建

    4

    生产数据测试 - 在生产数据副本上测试迁移，避免生产锁定

    输出示例

    Prisma 迁移工作流 
    # Prisma 迁移命令

# 从 schema 变更创建迁移
npx prisma migrate dev --name add_user_avatar

# 生产环境应用待处理迁移
npx prisma migrate deploy

# 重置数据库 (仅开发)
npx prisma migrate reset

# schema 变更后生成客户端
npx prisma generate

# 自定义 SQL 迁移 (Prisma 无法表达的操作)
npx prisma migrate dev --create-only --name add_email_index

# 编辑生成的 SQL 文件
# migrations/20240115_add_email_index/migration.sql
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_users_email ON users (email);
