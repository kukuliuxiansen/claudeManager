postgres-patterns

PostgreSQL数据库模式，涵盖查询优化、Schema设计、索引策略和安全最佳实践

类型: Pattern Skill

实现效果

postgres-patterns 是 PostgreSQL 数据库最佳实践的快速参考技能，基于 Supabase 团队的经验总结。提供查询优化、Schema设计、索引策略和安全配置的完整指南。

  • 索引策略速查表

    B-tree、GIN、BRIN索引类型选择指南，覆盖等值、范围、JSONB等查询模式

  • 数据类型最佳实践

    ID用bigint、字符串用text、时间用timestamptz、金额用numeric等规范

  • 常用模式参考

    复合索引顺序、覆盖索引、部分索引、UPSERT、游标分页、队列处理等模式

  • 反模式检测

    检测未索引外键、慢查询、表膨胀等常见问题

    • 核心原理

    PostgreSQL 优化的核心是正确选择索引类型和数据类型。根据查询模式选择合适的索引，使用正确的数据类型避免性能问题，并通过 RLS 策略保障数据安全。

    索引类型选择:

    • B-tree: 等值查询、范围查询 (默认索引类型)
    • GIN: JSONB、全文搜索、数组包含查询
    • BRIN: 时间序列范围数据、大表按顺序存储
    • 复合索引: 等值列在前，范围列在后

    数据类型规范:

    • ID使用 bigint，避免 int 和随机 UUID
    • 字符串使用 text，避免 varchar(255)
    • 时间使用 timestamptz，避免 timestamp
    • 金额使用 numeric(10,2)，避免 float

    使用方法

    postgres-patterns 适用于编写SQL查询、设计Schema、排查慢查询、实现RLS策略等场景。

// 常用模式示例

// 复合索引顺序 - 等值列在前，范围列在后
CREATE INDEX idx ON orders (status, created_at);
// 适用于: WHERE status = 'pending' AND created_at > '2024-01-01'

// 覆盖索引 - 避免回表查询
CREATE INDEX idx ON users (email) INCLUDE (name, created_at);
// 适用于: SELECT email, name, created_at FROM users

// 部分索引 - 只索引活跃数据
CREATE INDEX idx ON users (email) WHERE deleted_at IS NULL;

// UPSERT 模式
INSERT INTO settings (user_id, key, value)
VALUES (123, 'theme', 'dark')
ON CONFLICT (user_id, key)
DO UPDATE SET value = EXCLUDED.value;

// 游标分页 - O(1) 性能
SELECT * FROM products WHERE id > $last_id ORDER BY id LIMIT 20;

// 队列处理 - SKIP LOCKED
UPDATE jobs SET status = 'processing'
WHERE id = (
  SELECT id FROM jobs WHERE status = 'pending'
  ORDER BY created_at LIMIT 1
  FOR UPDATE SKIP LOCKED
) RETURNING *;

    使用技巧

    1

    索引顺序 - 等值条件列在前，范围条件列在后

    2

    避免 varchar - PostgreSQL 中 text 性能更好

    3

    游标分页 - 用 WHERE id > last_id 替代 OFFSET

    4

    RLS优化 - 在 USING 中用 SELECT auth.uid()

    5

    监控慢查询 - 启用 pg_stat_statements 扩展

    6

    安全默认 - REVOKE ALL ON SCHEMA public FROM public

## Anti-Pattern Detection Results

### Unindexed Foreign Keys
- orders.user_id → Add: CREATE INDEX idx ON orders (user_id)
- items.order_id → Add: CREATE INDEX idx ON items (order_id)

### Slow Queries (>100ms avg)
1. `SELECT * FROM orders WHERE user_id = ?` - Missing index
2. `SELECT * FROM products WHERE attributes @> '{}'` - Need GIN index

### Table Bloat
- orders: 5000 dead tuples - Run VACUUM ANALYZE
- logs: 10000 dead tuples - Consider autovacuum tuning
