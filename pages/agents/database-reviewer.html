数据库审查代理

专业审查数据库设计、查询性能和安全性的智能代理

类型: Agent类

实现效果

专注于数据库层审查的智能代理，诊断查询性能问题、优化查询执行计划并确保数据安全配置正确。

  • 查询性能优化
    诊断慢查询、缺少索引和full table scans，提供EXPLAIN ANALYZE指导
  • 架构设计审查
    评估表结构、数据类型选择、约束定义和规范化设计
  • 安全策略检查
    验证Row Level Security配置、权限设置和最小权限原则
  • 连接池配置
    审查连接池大小、超时设置和最大连接数配置
  • 并发控制审查
    诊断死锁风险、锁顺序不一致和长事务问题
  • 监控配置
    确保查询分析和性能跟踪已正确设置

    • 核心原理

    数据库审查代理采用SQL查询分析与数据库元数据检查相结合的双向审查策略。通过EXPLAIN ANALYZE诊断查询性能瓶颈，结合pg_stat_statements等系统视图检查查询模式，并审查Row Level Security策略和权限配置确保数据安全。

    使用方法

    触发场景与调用 
    # 触发场景
- 数据库迁移审查
- 查询性能优化需求
- 新功能上线前的数据库配置检查

# 在Claude Code中使用
claude -p "审查数据库设计和查询性能" --allowedTools "demo"

    使用技巧

    1
    所有WHERE/JOIN列必须建立索引，无例外
    2
    使用partial index处理软删除：WHERE deleted_at IS NULL
    3
    复合索引列顺序：等值条件在前，范围条件在后
    4
    覆盖索引使用INCLUDE(col)避免表查找
    5
    队列处理使用SKIP LOCKED提升10倍吞吐
    6
    分页使用游标而非OFFSET：WHERE id > $last

    输出示例

    Database Review Output 
    === CRITICAL 安全问题 ===
[CRITICAL] Row Level Security未启用
表: users
问题: multi-tenant表users缺少RLS策略
修复: `ALTER TABLE users ENABLE ROW LEVEL SECURITY;`

[CRITICAL] 使用SELECT *在生产环境
文件: src/repository/user_repo.ts:34
问题: `SELECT * FROM users WHERE id = ?` 返回不必要的列
修复: 显式指定所需列 `SELECT id, name, email FROM users...`

=== HIGH 查询性能 ===
[HIGH] 缺少索引导致全表扫描
查询: SELECT * FROM orders WHERE user_id = ?
问题: user_id列无索引
修复: `CREATE INDEX idx_orders_user_id ON orders(user_id);`

[HIGH] N+1查询模式
描述: 在循环中为每个项目执行单独查询
修复: 使用JOIN或IN子句批量获取数据

=== MEDIUM 最佳实践 ===
[MEDIUM] 使用OFFSET分页效率低下
查询: SELECT * FROM products LIMIT 10 OFFSET 100
修复: 使用游标分页 `WHERE id > $last_id ORDER BY id LIMIT 10`

[LOW] 使用int而非bigint存储主键
问题: id字段使用int类型
修复: 改用bigint防止ID耗尽

审查结论: 发现2个CRITICAL问题，3个HIGH问题，需要在生产部署前修复
