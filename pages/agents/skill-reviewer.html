技能审查代理

根据 plugin-dev 标准审查技能质量、内容组织和文档

类型: Agent类

实现效果

技能审查代理对 Claude Code 技能文件应用 plugin-dev 项目的质量标准，确保技能文档的质量、一致性和可用性。

  • 描述审查

    评估当前准确性、简洁性和可操作性

  • 内容质量

    评估 SKILL.md 的字数、风格和组织

  • 渐进式披露

    评估文档文件夹结构和层次

  • 严重问题识别

    标记 CRITICAL、MAJOR 和 MINOR 问题

    • 核心原理

    技能审查代理基于 plugin-dev 项目自身的质量标准，强调清晰、简洁和用户可发现性：

    • 描述优先 - 开头的描述必须准确和可操作
    • 渐进式披露 - 基本概念在 SKILL.md，高级主题在 references/
    • 可发现性 - 使用 moonlight search 兼容的关键词
    • 一致性 - 遵循 plugin-dev 的文档约定
    • 实用性 - 提供可运行的代码示例和命令

    使用方法

    触发时机与审查文件 
    // 触发时机
1. 新技能 PR 提交前
2. 技能维护和更新
3. 技能迁移到新 plugin 结构
4. 文档质量审查

// 被审查的文件
SKILL.md                    # 主文档（150-500 字最佳）
references/*.md            # 高级主题和深入文档
examples/*.md              # 代码示例
scripts/*.sh               # 工具脚本

```text

### Description Analysis
**Current:** [显示当前描述]
**Issues:**
- [问题 1]
**Recommendations:**
- [建议修复]
```

// 审查标准
| 项目 | 通过 | 需改进 | 严重 |
|-----|------|--------|------|
| 描述长度 | 150-500 字 | <100 或 >800 | - |
| 描述清晰性 | 可理解目标受众 | 模糊或冗余 | CRITICAL |
| 正确性 | 准确反映能力 | 遗漏关键信息 | CRITICAL |
| 使用示例 | 高质量代码示例 | 缺少或低质量 | HIGH |
| references 结构 | 逻辑组织 | 混乱或过多 | MEDIUM |

    使用技巧

    1

    短描述优先 - 描述应清晰简洁（200-300 字）

    2

    References 驱动 - 高级主题移到 references/

    3

    示例驱动 - 每个主要用例至少有一个示例

    4

    关键词优化 - 在描述中包含搜索关键词

    5

    新手友好 - 假设读者是该领域的初学者

    # 技能审查报告

## 概览
**技能名称**: git-commit-practice
**当前版本**: v1.2
**审查类型**: 质量审查

## 描述分析
**当前描述**: "Guidance for conventional commits and PR workflows"

**问题**:
- 过于宽泛，没有说明具体用途
- 未说明目标受众
- 缺少可操作的指导意见

**建议**:
- 明确指定适用场景
- 添加具体动作动词
- 包含预期结果

**改进后的描述**:
"`git-commit-practice` agent 在 PR 审查和预提交检查期间触发，确保所有提交符合 conventional commits 标准。审核提交消息格式、重写不清晰的提交，并建议适当的类型/范围/描述。"

## 内容质量
**SKILL.md 分析**:
- 当前字数: 3200 字 (**太长**)
- 建议范围: 150-500 字
- 写作风格: 专业但过于技术化
- 组织: TODO 列表过多，缺少清晰结构

**问题**:
- 定义太多而没有足够示例
- 没有新手友好的入门部分
- 缺少常见错误的快速参考

**建议**:
- 将详细定义移到 `references/concepts.md`
- 添加 "快速开始" 部分（5 分钟内可完成）
- 添加 `examples/fixup.md` 展示修复提交
- 使用表格整理常见类型

## 渐进式披露
**当前结构**:
- SKILL.md: 3200 字（主文档）
- references/: 4 个文件，共 8500 字
- examples/: 2 个文件
- scripts/: 3 个文件

**评估**:
[不是最佳] 太多主文档内容，references 过载

**建议**:
- 将 SKILL.md 限制到 300 字
- 移动 3 个定义文件到 `references/`
- 在 SKILL.md 中提供链接到 references
- 添加 `examples/` 文件夹中的常见场景

## 严重问题 (3)
#### CRITICAL (2)
- **SKILL.md:12**: 描述过于宽泛，无法理解目标
- **SKILL.md:45**: 缺少有效的使用示例

#### MAJOR (1)
- **SKILL.md:120**: TODO 列表Too long，未编号
- **references/advanced-concepts.md:1**: 冗长的技术定义

#### MINOR (2)
- **examples/pr-rewrite.md:23**: 注释过时
- **scripts/validate.sh:5**: 无 shebang 行

## 正面方面
- 项目结构组织良好
- references 文档全面
- 代码示例准确

## 整体评级
**需要改进** - 先解决 CRITICAL 问题

## 优先建议
1. **最高优先级**: 重写 SKILL.md 描述为 200-300 字
2. **第二优先级**: 添加至少 3 个新的使用示例
3. **第三优先级**: 重构文档结构，减少主文档字数
4. **第四优先级**: 更新过时的注释

## 边缘情况
- 技能描述良好但缺乏示例 → 建议添加更多代码示例
- 技能非常长（>5000 字）→ 强烈建议拆分到 references/
- 新技能（内容最少）→ 提供建设性构建指导
- 完美的技能 → 承认质量并建议微小改进
- 缺少参考文件 → 清晰报告路径错误
