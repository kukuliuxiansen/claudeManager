实现效果

类型设计分析代理审查代码中的类型设计，识别编码效率低下、不变量表达不良或缺乏编译时保证的类型。

  • 不变量识别

    找出类型必须保持的业务规则和约束

  • 封装质量评估

    评估类型是否正确隐藏内部实现

  • 不变量表达评估

    评分类型是否能够证明不变量在编译时成立

  • 改进建议

    提供具体的、可操作的类型设计改进建议

    • 核心原理

    类型设计分析代理基于键原则评估类型系统：

    • 编译时保证优于运行时检查 - 能在编译时证明的不要依赖运行时验证
    • 声明优于命令 - 类型定义应表达"是什么"而不是"怎么做"
    • 非法状态不可表示 - 类型应使无效状态无法编译
    • 构造函数验证至关重要 - 类型安全始于构造边界
    • 不可变性简化维护 - 避免状态突变保持不变量

## Type: UserEmail

### Invariants Identified
- 邮箱格式应符合 RFC 5322
- 邮箱不应为 null 或 undefined
- 邮箱长度应在 3-254 字符之间

### Ratings
- **Encapsulation**: 3/10
  邮箱是公共字段，可被任意赋值，包括无效格式
- **Invariant Expression**: 2/10
  类型只是 `string`，没有编译时保证
- **Invariant Usefulness**: 8/10
  邮箱格式在业务中很重要（身份验证、通知）
- **Invariant Enforcement**: 1/10
  仅在运行时验证，无编译时保证

### Strengths
- 捕获运行时格式错误
- 有清晰的业务目的

### Concerns
- `new UserEmail("invalid")` 编译通过但运行时失败
- 公共字段允许绕过验证方法
- 没有类型为特定用途（如 "email" vs "username"）

### Recommended Improvements
1. **使用 Brand Type 获取编译时检查**
```typescript
// 推荐的类型定义
interface EmailBrand {
  readonly __emailBrand: unique symbol;
}

type Email = string & EmailBrand;

// 工厂函数进行验证
export function createEmail(value: string): Email {
  if (!validateEmail(value)) {
    throw new Error('Invalid email format');
  }
  return value as Email;
}
```

2. **隐藏实现细节**
```typescript
// 将邮箱设为私有，通过 getter 访问
class User {
  private readonly email: Email;

  constructor(email: string) {
    this.email = createEmail(email);
  }

  getEmail(): Email {
    return this.email;
  }
}
```

3. **添加单位验证测试**
```typescript
test('createEmail rejects invalid formats', () => {
  expect(() => createEmail('invalid')).toThrow();
  expect(() => createEmail('')).toThrow();
});
```

## Type: OrderStatus

### Invariants Identified
- 状态必须是预定义集合之一
- 状态转换应遵循有限自动机

### Ratings
- **Encapsulation**: 9/10
  状态是私有的，只能通过方法修改
- **Invariant Expression**: 8/10
  使用联合类型限制可能的值
- **Invariant Usefulness**: 9/10
  状态转换 correctness 很重要
- **Invariant Enforcement**: 9/10
  编译时限制有效状态

### Strengths
- 良好的类型定义
- 良好的封装
- 合理的验证

### Concerns
- `cancel()` 方法可能应该验证当前状态
- 可以添加状态转换图作为文档

### Recommended Improvements
1. **添加状态转换验证**
```typescript
type OrderStatus = 'pending' | 'confirmed' | 'shipped' | 'delivered';

class Order {
  private status: OrderStatus;

  confirm() {
    if (this.status !== 'pending') {
      throw new Error('Cannot confirm non-pending order');
    }
    this.status = 'confirmed';
  }

  ship() {
    if (this.status !== 'confirmed') {
      throw new Error('Cannot ship non-confirmed order');
    }
    this.status = 'shipped';
  }
}
```

- **强类型定义是好的实践**
- **而 Email 类型需要改进**
- 优先处理编译时保证（如 OrderStatus）
