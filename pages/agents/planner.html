规划代理

复杂功能实现的规划专家

类型: Agent类

实现效果

分析需求并创建详细实现计划的专家代理，将复杂功能分解为可管理的步骤。

  • 需求分析
    完整理解功能请求，识别成功标准和约束条件
  • 架构审查
    分析现有代码库结构，识别受影响组件
  • 步骤分解
    将复杂功能分解为清晰可执行的具体步骤
  • 优先级排序
    根据依赖关系和风险确定最佳实施顺序

    • 核心原理

    规划代理采用四阶段工作流程：

    • 阶段1：需求分析：理解功能请求，识别成功标准和约束条件
    • 阶段2：架构审查：分析现有代码库结构，识别受影响组件
    • 阶段3：步骤分解：创建详细的步骤清单，包含文件路径和依赖关系
    • 阶段4：实施排序：根据依赖关系和可测试性确定最佳实施顺序

    使用方法

    触发方式 
    调用场景：需要详细实现计划的功能开发请求

触发条件：
- 新功能开发 requests
- 重构任务规划
- 复杂 bug 修复方案
- 大型特性拆解

所需信息：
1. 功能描述和业务目标
2. 成功标准和约束条件
3. 现有代码库上下文
4. 技术栈和框架信息

    使用技巧

    1
    确保每个步骤都有明确的文件路径和具体操作
    2
    步骤之间要明确定义依赖关系和风险评估
    3
    每个阶段应该可以独立交付和验证
    4
    考虑边缘情况和潜在的错误场景
    5
    优先选择扩展现有代码而非重写
    6
    规划中必须包含测试策略和覆盖率目标

    输出示例

    Implementation Plan Output 
    ## 实现计划：Stripe 订阅计费

### 概述
添加订阅计费功能，支持免费/专业/企业套餐，用户通过Stripe结账，

### 需求
- 三个套餐: Free（默认）、Pro（$29/月）、Enterprise（$99/月）
- Stripe结账流程
- webhook处理订阅生命周期
- 基于订阅套餐的功能限制

### 架构变更
- 新表: `subscriptions` (user_id, stripe_customer_id, stripe_subscription_id, status, tier)
- 新API: `app/api/checkout/route.ts`
- 新API: `app/api/webhooks/stripe/route.ts`
- 新中间件: tier检查
- 新组件: `PricingTable`

### 实施步骤

#### 阶段1: 数据库和后端 (2文件)
1. 创建订阅迁移 (supabase/migrations/004_subscriptions.sql)
   - 创建表和RLS策略
   - 依赖: 无
2. 创建Stripe webhook处理 (src/app/api/webhooks/stripe/route.ts)
   - 处理取消/更新等事件
   - 依赖: 步骤1

#### 阶段2: 结账流程 (2文件)
3. 创建结账API (src/app/api/checkout/route.ts)
   - 创建Stripe结账会话
   - 依赖: 步骤1
4. 构建价格页面 (src/components/PricingTable.tsx)
   - 显示三个套餐
   - 依赖: 步骤3

#### 阶段3: 功能限制 (1文件)
5. 添加套餐检查中间件 (src/middleware.ts)
   - 检查用户套餐权限
   - 依赖: 步骤1-2

### 测试策略
- 单元测试: webhook解析、套餐检查
- 集成测试: 结账会话创建
- E2E测试: 完整升级流程

### 风险
- webhook事件乱序 → 使用时间戳、幂等更新
- 用户升级但webhook失败 → 轮询Stripe作为后备
