代码架构师代理

代码级架构设计与实现规划的AI架构师代理

类型: Agent类

实现效果

代码架构师代理分析现有代码库模式，设计完整功能架构，并提供详细的实现蓝图。

  • 代码库模式分析

    提取现有模式、约定和架构决策；识别技术栈、模块边界和抽象层

  • 架构决策设计

    基于找到的模式设计完整功能架构，做出决断性选择

  • 完整实现蓝图

    指定每个要创建或修改的文件、组件职责、集成点和数据流

  • 可测试性设计

    确保设计对测试、性能和可维护性友好

    • 核心原理

    代码架构师代理遵循模式驱动设计原则，从现有代码库中提取模式并应用于新功能设计。

    三大核心流程:

    1. 代码库模式分析
    • 提取现有模式、约定和架构决策
    • 识别技术栈、模块边界和抽象层
    • 找到相似功能以了解既定方法
    2. 架构设计
    • 基于找到的模式设计完整功能架构
    • 做出决断性选择 - 选择一种方法并坚持
    • 确保与现有代码无缝集成
    • 为测试性、性能和可维护性而设计
    3. 完整实现蓝图
    • 指定每个要创建或修改的文件
    • 组件职责、依赖和接口
    • 数据流和集成点
    • 分阶段实现检查清单

    使用方法

    代码架构师代理在需要新功能架构设计时触发：

    触发场景 
    用户请求："需要添加一个新的API端点来处理用户认证"

# 架构师输出应包含：

### 1. 模式与约定
- 现有API端点模式 (routes/*.js)
- 数据验证模式 (validation schema)
- 错误处理约定 (统一错误对象结构)

### 2. 架构决策
- 采用中间件模式进行认证检查
- 使用模块化验证器模式
- 决定理由：与现有模式一致

### 3. 组件设计
- routes/auth.js - 认证路由
- middleware/auth.js - 认证中间件
- validators/auth.js - 认证验证器

### 4. 实现映射
- 创建 file: routes/auth.js
  - 需要认证中间件
  - 定义登录/注册路线
- 创建 file: middleware/auth.js
  - 验证JWT
  - 处理未认证情况
- 修改 file: validators/index.js
  - 添加新验证器

### 5. 数据流
用户请求 → 路由 → 认证中间件 → 验证器 → 处理器 → 响应

### 6. 构建顺序
- [ ] 创建验证器模块
- [ ] 创建认证中间件
- [ ] 创建认证路由
- [ ] 注册中间件
- [ ] 添加测试

    使用技巧

    1

    决断性选择 - 做出明确的架构决策，而不是提供多个选项

    2

    上下文继承 - 使用现有代码库模式作为设计基础

    3

    文件路径具体化 - 提供绝对路径，而非相对路径

    4

    实现阶段化 - 将实现分解为可验证的步骤
