开发舰队命令

多Agent编排命令，用于大规模项目开发的自动化任务编排

类型: Command类

实现效果

通过Multi-Agent编排，自动将用户描述分解为链接的任务序列，实现大规模项目的自动化开发。

  • 项目计划生成
    AI自动将用户描述分解为链接的Mission DAG依赖图
  • 自动任务分发
    依赖完成的Mission自动触发后续Mission（auto_dispatch=true）
  • 自动合并工作
    每个Agent完成后自动将其工作tree合并到主分支
  • 结构化报告生成
    每个Mission完成后生成文件变更、错误等结构化报告
  • 并发任务控制
    可配置并发数量（默认3），多余任务排队等待空闲槽位

    • 核心原理

    开发舰队命令基于DAG（有向无环图）依赖编排模型，实现自动化任务流水线：

    1. 计划阶段：用户描述项目 → AI分解为Mission DAG → 显示计划并获取批准
    2. 分发阶段：dispatch_mission(M1) → Agent在worktree中运行 → 自动合并
    3. 流水线阶段：M1完成后 → M2自动触发（depends_on M1）→ 循环直至所有Mission完成
    4. 报告阶段：get_report(M) → 返回files_modified、errors、next_steps等

    核心是通过auto_dispatch机制和depends_on依赖实现任务的自动流水线执行。

    使用方法

    触发场景与调用 
    触发场景：
- 复杂项目规划和执行
- 大规模重构任务
- 多阶段功能开发
- 代码库迁移任务

调用方式：
在Claude Code中输入：
/devfleet

使用流程：
1. plan_project(prompt) - 生成Mission DAG
2. 显示计划并等待批准
3. dispatch_mission(mission_id) - 启动第一个Mission
4. 剩余Mission自动按依赖顺序触发
5. get_report(mission_id) - 获取每个Mission的报告

    使用技巧

    1
    先批准计划：除非用户说"继续"，否则显示计划后等待批准
    2
    检查仪表板：使用get_dashboard()查看可用槽位和队列状态
    3
    轮询而非阻塞：对于长时间任务，使用get_mission_status轮询而非wait_for_mission
    4
    检查错误报告：如果Mission失败，读取报告了解错误前重试
    5
    避免循环依赖：依赖形成DAG，绝不创建循环依赖

### 流程
```
用户描述项目
  → plan_project(prompt) → mission DAG with dependencies
  → 显示计划，获取批准
  → dispatch_mission(M1) → Agent spawns in worktree
  → M1完成 → auto-merge → M2自动触发(依赖M1)
  → M2完成 → auto-merge
  → get_report(M2) → files_changed, what_done, errors, next_steps
  → 向用户报告摘要
```

### 工作流
1. **计划项目** 从用户描述:
```
mcp__devfleet__plan_project(prompt="<用户描述>")
```
返回带有链接Mission的项目。向用户显示：
- 项目名称和ID
- 每个Mission：标题、类型、依赖
- 依赖DAG（哪些Mission阻塞哪些）

2. **等待用户批准** 后分发。清晰显示计划。

3. **分发第一个Mission**（取决于on为空的）:
```
mcp__devfleet__dispatch_mission(mission_id="<first_mission_id>")
```
剩余Mission在其依赖完成时自动分发（因为plan_project创建时设置了auto_dispatch=true）。

4. **监控进度** - 检查运行状态:
```
mcp__devfleet__get_dashboard()
或
mcp__devfleet__get_mission_status(mission_id="<id>")
```

5. **读取报告** 每个完成的Mission:
```
mcp__devfleet__get_report(mission_id="<mission_id>")
```
报告包含：files_changed, what_done, what_open, what_tested, what_untested, next_steps, errors_encountered。

### 可用工具列表
| 工具 | 用途 |
|------|------|
| `plan_project(prompt)` | AI将描述分解为链接的Mission，auto_dispatch=true |
| `create_project(name, path?, description?)` | 手动创建项目，返回project_id |
| `create_mission(project_id, title, prompt, depends_on?, auto_dispatch?)` | 添加Mission |
| `dispatch_mission(mission_id, model?, max_turns?)` | 启动Agent |
| `cancel_mission(mission_id)` | 停止运行的Agent |
| `wait_for_mission(mission_id, timeout_seconds?)` | 阻塞直到完成 |
| `get_mission_status(mission_id)` | 检查进度不阻塞 |
| `get_report(mission_id)` | 读取结构化报告 |
| `get_dashboard()` | 系统概览 |
| `list_projects()` | 浏览项目 |
| `list_missions(project_id, status?)` | 列出Mission |

### 指导原则
- 除非用户说"继续"，否则始终确认计划
- 报告状态时包括Mission标题和ID
- 如果Mission失败，在重试前读取报告了解错误
- Agent并发可配置（默认:3）。多余Mission排队并自动分发
- 依赖形成DAG - 永远不要创建循环依赖
- Agent在完成时自动合并其工作tree。如果合并冲突，更改保留在worktree分支供手动解决。
