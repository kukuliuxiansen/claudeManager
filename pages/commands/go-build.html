Go构建修复

自动诊断并修复Go构建错误

类型: Command类

实现效果

运行全面的Go诊断工具，逐个修复构建错误直至项目成功编译：

  • 多层诊断扫描
    go build、go vet、staticcheck、golangci-lint 全面检查
  • 错误分组分类
    按文件分组错误，按严重性排序（编译错误 > Vet > Lint）
  • 增量修复验证
    每次修复后重新运行构建，确保修复有效且不引入新错误

    • 核心原理 

    
    诊断命令执行流程
    

    
Step 1: 运行诊断
├─ go build ./...    # 编译错误（最高优先级）
├─ go vet ./...      # 潜在问题
├─ staticcheck ./... # 静态分析
└─ golangci-lint run # 代码风格

Step 2: 解析错误输出
├─ 按文件分组错误
├─ 提取错误行号和描述
└─ 按严重性排序

Step 3: 逐个修复（每次修复后验证）
├─ 读取出错文件
├─ 分析错误原因
├─ 最小化修改
└─ 重新运行构建

Step 4: 验证修复
├─ go build ./...    # 确保没有新错误
├─ go vet ./...      # 确保没有新警告
└─ go test ./...     # 最终验证

    使用方法

    触发场景与调用 
    # 当 go 构建失败时触发

"/go-build"
格式: 修复 Go 构建错误
要求: 运行 /go-build 开始诊断和修复

示例用法：
1. 某次构建失败后运行
2. more than one package in directory at ...
3. undefined: X  # 未定义的变量或函数

关键点：
- 命令会自动运行所有诊断工具
- 逐个修复错误并验证
- 如果 3 次尝试后仍失败，会询问用户

    使用技巧

    1
    先运行 `go build` 修复编译错误，再处理 vet 和 lint 警告
    2
    一次只修复一个错误，避免引入连锁问题
    3
    使用 `go mod tidy` 修复依赖问题，而不是手动修改 go.mod
    4
    未使用的导入使用 `goimports -w` 自动修复
    5
    循环依赖错误需要重新设计包结构
    6
    软件包未找到时，先运行 `go get` 再构建

 Step 2: 修复中
#[1/3] 未定义的标识符
文件：internal/service/user.go:25
错误：undefined: UserRepository
修复：添加缺失的 import 语句
```go
import "project/internal/repository"
```
结果：通过

#[2/3] 类型不匹配
文件：internal/handler/api.go:42
错误：cannot use x (type string) as type int
修复：添加类型转换
```go
countStr := params.Get("count")
count, _ := strconv.Atoi(countStr)
```
结果：通过

#[3/3] 缺失返回语句
文件：internal/handler/api.go:58
错误：missing return at end of function
修复：添加返回语句
```go
return user, nil
```
结果：通过
