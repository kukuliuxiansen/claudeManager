功能开发

系统化功能开发流程：探索-设计-实现-审查

类型: Command类

实现效果

提供完整的功能开发工作流，确保从需求理解到代码实现的每个环节都经过充分考虑和验证：

  • 系统化七阶段流程
    Discovery → Codebase Exploration → Clarifying Questions → Architecture Design → Implementation → Quality Review → Summary
  • 提问确认机制
    每个关键阶段都需用户确认，避免误解和返工
  • 代码审查整合
    自动并行启动多个审查代理，从不同角度确保代码质量

    • 核心原理 

    
    七阶段开发流程
    

    
Phase 1: Discovery
├─ 创建任务清单
├─ 理解需求（问题、功能、约束）
└─ 用户确认理解正确

Phase 2: Codebase Exploration
├─ 并行启动2-3个探索代理
├─ 各自关注不同方面（架构/模式/扩展点）
└─ 汇总关键文件和实现模式

Phase 3: Clarifying Questions
├─ 分析模糊和边界情况
├─ 列出所有问题供用户确认
└─ 等待答案后再继续设计

Phase 4: Architecture Design
├─ 并行启动2-3个架构设计代理
├─ 提出不同权衡方案
└─ 用户选择实现方式

Phase 5: Implementation
├─ 等待用户批准
├─ 读取相关文件
└─ 按照选择的架构实现

Phase 6: Quality Review
├─ 并行启动3个审查代理
├─ 检查：简洁性/正确性/约定
└─ 用户决定是否修复

Phase 7: Summary
├─ 标记所有任务完成
├─ 总结实现内容
└─ 记录关键决策

    使用方法

    触发场景与调用 
    # 当你说 /feature-dev 或明确要求功能开发时触发

"/feature-dev"
格式: 实现一个[功能描述]
要求: 使用/feature-dev命令开始系统化开发流程

示例用法：
1. /feature-dev 我需要一个用户认证模块
2. /feature-dev 添加支付处理功能，支持微信和支付宝

关键点：
- 需要清晰描述功能需求
- 命令会引导你完成所有阶段
- 每个关键点都会等待你的确认

    使用技巧

    1
    Phase 3（提问确认）最重要 - 不要跳过，这能避免后续大量返工
    2
    Phase 2探索时，确保代理关注不同方面，避免重复工作
    3
    Phase 4架构设计，明确告诉代理你的优先级（速度 vs 质量）
    4
    Phase 6审查时，关注 Critical 和 High 严重级问题
    5
    如果有不确定的需求，直接问："这个功能应该做什么？"
    6
    使用 TodoWrite 跟踪所有阶段进度，保持清晰

Phase 1: Discovery
我们需要实现：用户认证模块
问题确认：
1. 用户认证需要包含哪些功能？（登录/注册/忘记密码/第三方登录）
2. 认证方式：email/password、手机号验证码、还是第三方（微信/支付宝）？
3. 会话管理：JWT 还是 Session？
4. 安全要求：密码复杂度、登录失败限制、两步验证

Phase 2: Codebase Exploration
探索代理已启动（3个并行）：
- 代理1：类似功能实现（参考 project/auth）
- 代理2：项目架构分析（查看 backend/auth/）
- 代理3：UI 模式检测（查看 frontend/pages/auth/）

Phase 3: Clarifying Questions
需要确认：
- 会话有效期：24小时还是可配置？
- 密码存储：bcrypt 还是 argon2？
- 登录失败锁定：3次失败锁定5分钟，还是永久？

Phase 4: Architecture Design
设计建议（3种方案）：
方案A（最小改动）：在现有 auth 模块扩展现有功能
方案B（清洁架构）：创建独立的 auth 聚合，清晰分离 concerns
方案C（实用平衡）：渐进式重构，先实现再优化

Phase 5: Implementation
批准后开始实现...

Phase 6: Quality Review
审查代理并行启动（3个）：
- 简洁性审查：代码 DRY 原则
- 正确性审查：功能实现准确性
- 约定审查：是否符合项目规范

Phase 7: Summary
已完成：
- 用户认证模块
- API 端点 /auth/login, /auth/register
- 文件修改：backend/auth/, frontend/pages/auth/
- 测试覆盖：85%
