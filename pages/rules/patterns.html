常见模式规则

定义骨架项目策略、设计模式、通信模式和调试流程

类型: 通用规则

骨架项目策略

实现新功能时：

骨架项目流程 
1. 搜索久经考验的骨架项目
2. 使用并行代理评估选项：
   - 安全评估
   - 可扩展性分析
   - 相关性评分
   - 实现规划
3. 克隆最佳匹配作为基础
4. 在已验证的结构内迭代

设计模式

Repository 模式

将数据访问封装在一致的接口后：

  • 定义标准操作：findAll, findById, create, update, delete
  • 具体实现处理存储细节（数据库、API、文件等）
  • 业务逻辑依赖抽象接口，而非存储机制
  • 支持轻松切换数据源，简化 mock 测试

API 响应格式

为所有 API 响应使用一致的封装：

  • 包含成功/状态指示器
  • 包含数据载荷（错误时可为空）
  • 包含错误消息字段（成功时可为空）
  • 分页响应包含元数据（total, page, limit）

通信模式

列出项目：

当被要求"列出所有 X"时，在一个响应中显示完整列表，而不是分批显示。

如果完整列表太长：

  • 首先明确总数
  • 询问用户是否想查看所有项目

原因：分批列出会导致用户沮丧，因为他们实际上想要的是缩减，而不是分段显示的完整列表。

调试流程

根因验证：

当用户报告 bug 或问题时：

调试步骤 
1. 首先在正确的组件中验证根因
2. 在修改前端代码前检查后端数据存储
3. 不要做假设并修改错误的代码库

原因：对错误组件进行大量更改会浪费时间并产生不必要的混乱。

使用技巧

1

骨架优先 - 寻找久经考验的项目作为基础

2

Repository 模式 - 封装数据访问接口

3

一次性列出 - 不要分批列出项目

4

验证根因 - 在正确组件中调试
