编码风格规则

定义不可变性原则、文件组织、错误处理和代码质量标准

类型: 通用规则

关键规则：不可变性

始终创建新对象，永远不要修改现有对象。不可变数据防止隐藏的副作用，使调试更容易，并支持安全并发。

实现效果

编码风格规则确保代码质量、可维护性和一致性。

  • 不可变性

    始终创建新对象，永不修改现有对象

  • 文件组织

    多小文件优于少大文件，按功能/领域组织

  • 错误处理

    全面处理错误，永不静默吞掉

  • 输入验证

    在系统边界验证所有输入

    • 核心原理

    不可变性示例：

    伪代码对比 
    // 错误方式
WRONG:  modify(original, field, value) → changes original in-place

// 正确方式
CORRECT: update(original, field, value) → returns new copy with change

    文件组织原则：

    • 高内聚、低耦合
    • 典型文件 200-400 行，最多 800 行
    • 从大模块提取工具函数
    • 按功能/领域组织，而非按类型

    代码质量检查清单

    标记工作完成前检查：

    • 代码可读且命名良好
    • 函数小巧（<50 行）
    • 文件专注（<800 行）
    • 无深层嵌套（>4 层）
    • 正确的错误处理
    • 无硬编码值（使用常量或配置）
    • 无突变（使用不可变模式）

    使用技巧

    1

    不可变优先 - 创建新对象而非修改现有

    2

    小文件 - 200-400 行，最多 800 行

    3

    错误显式 - 每层显式处理，永不静默

    4

    边界验证 - 系统边界验证所有输入
