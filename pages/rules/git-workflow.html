Git 工作流规则

定义提交消息格式、Pull Request 流程和分支策略

类型: 通用规则

实现效果

Git 工作流规则确保一致的提交历史和清晰的 Pull Request 流程。

  • 提交消息格式

    遵循 conventional commits 规范

  • PR 工作流

    分析完整提交历史，包含测试计划

  • 完整开发流程

    与 development-workflow.md 配合使用

    • 核心原理

    提交消息格式：

    Conventional Commits 
    <type>: <description>

<optional body>

    提交类型：

    feat fix refactor docs test chore perf ci

    PR 工作流

    创建 PR 时：

    Pull Request 流程 
    1. 分析完整提交历史（不仅是最新的提交）
2. 使用 git diff [base-branch]...HEAD 查看所有更改
3. 起草全面的 PR 摘要
4. 包含测试计划和 TODOs
5. 如果是新分支，使用 -u 标志推送

    使用技巧

    1

    类型明确 - 使用正确的提交类型前缀

    2

    完整分析 - 分析所有提交，不仅最新

    3

    测试计划 - PR 中包含测试计划和 TODO

    4

    新分支推送 - 使用 -u 标志
